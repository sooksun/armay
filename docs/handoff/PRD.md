# PRD — Rental Broker Finance Control System

> ระบบควบคุมการรับเงิน–จ่ายเงินสำหรับนายหน้าปล่อยเช่าห้องพัก บ้านพัก แฟลต และคอนโด
> เอกสารนี้คือ "อะไร & ทำไม" — สถาปัตยกรรมอยู่ใน [`plan.md`](./plan.md), งานย่อยอยู่ใน [`tasks.md`](./tasks.md)

---

## 1. Problem

นายหน้าปล่อยเช่าอสังหาริมทรัพย์รับเงินจากผู้เช่าหลายห้อง/หลายอาคาร แล้วต้องส่งต่อให้เจ้าของแต่ละราย หลังหักค่านายหน้า ค่าทำความสะอาด และค่าซ่อม ปัจจุบันจดในสมุด/Excel/แชท ทำให้ **ลืมเก็บเงิน จ่ายซ้ำ จ่ายผิดเจ้าของ ตรวจย้อนหลังไม่ได้ และไม่รู้กำไรสุทธิ** ยิ่งจำนวนห้องมากยิ่งพลาด และไม่มีหลักฐานผูกกับธุรกรรมอย่างเป็นระบบ

## 2. Goals

- **G1** — ตอบได้ทันทีว่า "เงินก้อนนี้มาจากใคร ห้องไหน ต้องส่งเจ้าของคนใด หักอะไรไปแล้ว เหลือสุทธิเท่าไร มีหลักฐานไหม"
- **G2** — บริหารเจ้าของ อาคาร ห้อง ผู้เช่า สัญญาเช่า รายรับ รายจ่าย และการจ่ายเงินคืนเจ้าของ ในที่เดียว โดยยึด "ห้อง" และ "รายการเช่า" เป็นแกนกลาง
- **G3** — ทุกธุรกรรมมีสถานะตรวจสอบ แนบหลักฐานได้ และมี Audit Log ย้อนหลัง
- **G4** — Dashboard + รายงานสรุปรายรับ/รายจ่าย/ยอดค้างรับ/ยอดค้างจ่าย/กำไรเบื้องต้น พร้อมตัวกรองและ export Excel
- **G5** — ป้องกันความผิดพลาดเชิงเงิน: กันบันทึกรายรับซ้ำ กันจ่ายเจ้าของซ้ำ และล็อกรายการที่อนุมัติแล้ว

## 3. Non-goals (สำหรับ MVP v1)

- **NG1** — ไม่มี OCR อ่านสลิป / import bank statement / กระทบยอดอัตโนมัติ (→ v2)
- **NG2** — ไม่มี Owner Portal / Tenant Portal (เจ้าของ/ผู้เช่าล็อกอินเอง) — แต่ **ออกแบบ data model เผื่อไว้**
- **NG3** — ไม่มีแจ้งเตือนผ่าน LINE / ออกใบแจ้งหนี้ / ใบเสร็จ / คำนวณภาษี / ระบบ ticket งานซ่อม / ปฏิทินห้องว่าง (→ v2)
- **NG4** — ไม่มี 2FA (ออกแบบเผื่อ แต่ไม่ทำใน v1)
- **NG5** — ไม่รองรับหลายบริษัท/หลาย tenant (single-tenant business ของนายหน้ารายเดียว)

## 4. Target users & JTBD

| Persona | Job-to-be-done | Current workaround | บทบาทในระบบ |
|---|---|---|---|
| **นายหน้า / เจ้าของกิจการ** (Admin) | คุมภาพรวมเงินเข้า-ออก อนุมัติธุรกรรม จ่ายเจ้าของให้ถูก และรู้กำไร | สมุด/Excel/แชท | `ADMIN` — ทำได้ทุกอย่าง + อนุมัติ + ตั้งค่า |
| **ผู้ช่วย / พนักงาน** (Staff) | บันทึกผู้เช่า สัญญา รายรับ รายจ่าย และแนบสลิป | จดใส่ Excel ส่งให้เจ้านาย | `STAFF` — บันทึกได้ แต่ห้ามลบ/แก้รายการที่อนุมัติแล้ว |

## 5. User stories

- **US-01** — ในฐานะ Admin ฉันต้องการเพิ่มเจ้าของและผูกกับหลายห้อง เพื่อรู้ว่าเจ้าของแต่ละรายมีห้องใดบ้าง
  - **Given** ยังไม่มีเจ้าของรายนี้ **When** เพิ่มเจ้าของแล้วสร้างห้องผูก owner **Then** หน้าเจ้าของแสดงห้องทั้งหมดของเขา
- **US-02** — ในฐานะ Staff ฉันต้องการสร้างรายการเช่าจากห้องที่ว่าง เพื่อเริ่มเก็บเงินผู้เช่า
  - **Given** ห้องสถานะ `AVAILABLE` **When** สร้าง contract เลือกผู้เช่า+ช่วงวันที่ **Then** ระบบ auto-fill ค่าเช่า/ประกัน/ทำความสะอาดจาก default ห้อง และคำนวณยอดรวม
- **US-03** — ในฐานะ Staff ฉันต้องการบันทึกรับเงินพร้อมแนบสลิป เพื่อให้รู้ว่าผู้เช่าจ่ายครบหรือยัง
  - **Given** contract ที่ยังค้างชำระ **When** บันทึกรายรับ + แนบสลิป **Then** สถานะเป็น `PENDING` รอ Admin ตรวจ และยอดค้างลดลง
- **US-04** — ในฐานะ Admin ฉันต้องการอนุมัติรายรับ เพื่อยืนยันว่าเงินเข้าจริง
  - **Given** รายรับ `PENDING` **When** กดอนุมัติ **Then** สถานะเป็น `VERIFIED` และล็อกห้ามแก้โดยตรง
- **US-05** — ในฐานะ Admin ฉันต้องการให้ระบบคำนวณยอดจ่ายเจ้าของจากรายรับหักค่าใช้จ่าย เพื่อจ่ายให้ถูกและไม่ซ้ำ
  - **Given** ห้องที่มีรายรับ VERIFIED และค่าใช้จ่ายที่หักเจ้าของ **When** กดคำนวณ payout **Then** ระบบแสดง breakdown (ที่มาการหักทุกบรรทัด) และยอดสุทธิ กันไม่ให้ดึงธุรกรรมเดิมมาจ่ายซ้ำ
- **US-06** — ในฐานะ Staff ฉันต้องการบันทึกค่าแม่บ้าน/ค่าซ่อมผูกกับห้อง พร้อมรูปก่อน-หลัง เพื่อรู้ต้นทุนรายห้อง
- **US-07** — ในฐานะ Admin ฉันต้องการ Dashboard + รายงานที่กรองตามเดือน/อาคาร/เจ้าของ/ผู้เช่า/ห้อง และ export Excel

## 6. Functional requirements

> ทุก FR ต้อง map ไป ≥1 task ใน `tasks.md` (ดูตาราง traceability ท้ายไฟล์)

### Auth & Users
- **FR-001** — ระบบต้องให้ผู้ใช้ login ด้วย email + password (hash ด้วย bcrypt) และออก session (JWT httpOnly cookie)
- **FR-002** — ระบบต้องบังคับสิทธิ์ 3 ระดับ (`ADMIN`, `STAFF`, `VIEWER`) ที่ฝั่ง server ทุก endpoint
- **FR-003** — Admin ต้องจัดการผู้ใช้ (สร้าง/แก้/ปิดใช้งาน/กำหนด role) ได้

### เจ้าของ / อาคาร / ห้อง / ผู้เช่า / บัญชี (Master Data)
- **FR-010** — จัดการเจ้าของ (CRUD) พร้อมข้อมูลธนาคาร/PromptPay และดูห้องในความดูแล + ยอดที่ต้องจ่าย
- **FR-011** — จัดการอาคาร/โครงการ (CRUD) รองรับหลายประเภท (คอนโด/แฟลต/บ้านพัก/อาคาร/หอพัก ฯลฯ)
- **FR-012** — จัดการห้อง/ยูนิต (CRUD) ต้องผูกกับ **อาคาร** และ **เจ้าของ** เสมอ กำหนดค่าเช่า/ประกัน/ทำความสะอาด/ค่านายหน้า default และสถานะห้อง
- **FR-013** — จัดการผู้เช่า (CRUD) + ประวัติการเช่า + ยอดค้างชำระ + Blacklist/Watchlist
- **FR-014** — จัดการบัญชีรับ–จ่าย (CRUD): ธนาคาร/เลขบัญชี/PromptPay/QR/เงินสด แยกประเภทบัญชี
- **FR-015** — ทุก entity หลักต้องมีรหัสอัตโนมัติ (`owner_code`, `room_code`, ...) ที่ unique และกัน race

### สัญญาเช่า / รายการจอง
- **FR-020** — สร้าง/แก้/list รายการเช่า ต้องผูก **ผู้เช่า + ห้อง** (และ owner/property ที่ derive จากห้อง)
- **FR-021** — เมื่อสร้าง contract ต้อง auto-fill ค่า default จากห้อง และคำนวณ `total_amount` = ค่าเช่า + ประกัน + ทำความสะอาด + อื่นๆ − ส่วนลด
- **FR-022** — contract มีสถานะการเช่า (`BOOKED/CHECKED_IN/ENDED/CANCELLED`) และสถานะชำระเงิน (`UNPAID/PARTIAL/PAID/OVERDUE`) ปิด/ยกเลิกได้พร้อมเหตุผล

### รายรับ
- **FR-030** — บันทึกรายรับหลายรายการต่อ 1 contract แยกประเภท (ค่าเช่า/ประกัน/ทำความสะอาด/น้ำ/ไฟ/ค่าปรับ/อื่นๆ) เลือกช่องทาง + บัญชีที่รับ + เลขอ้างอิง
- **FR-031** — แนบหลักฐาน (สลิป/รูป/ใบเสร็จ) กับรายรับได้
- **FR-032** — ระบบต้องเตือนรายการซ้ำเมื่อ `income_date + amount + payment_method + transaction_reference` ตรงกัน (override ได้เฉพาะ Admin)
- **FR-033** — คำนวณยอดค้างชำระของ contract อัตโนมัติจากรายรับที่ VERIFIED เทียบ `total_amount`

### รายจ่าย (ค่าใช้จ่ายห้อง / แม่บ้าน / ช่าง)
- **FR-040** — บันทึกรายจ่ายผูกกับ **ห้อง** เสมอ (และ contract ถ้ามี) แยกประเภท (ทำความสะอาด/ซ่อม/วัสดุ/น้ำ/ไฟ/เน็ต/ส่วนกลาง/เดินทาง/บริหาร/โฆษณา/อื่นๆ)
- **FR-041** — ระบุผู้รับผิดชอบค่าใช้จ่าย: `BROKER` (นายหน้า) / `OWNER` (เจ้าของ) / `TENANT` (ผู้เช่า)
- **FR-042** — แนบสลิป + รูปก่อนซ่อม + รูปหลังซ่อม และแสดงประวัติซ่อม/ทำความสะอาดรายห้อง
- **FR-043** — หน้า "งานแม่บ้าน/งานซ่อม" เป็นมุมมองของรายจ่ายประเภท `CLEANING`/`REPAIR`

### จ่ายเงินเจ้าของ
- **FR-050** — ระบบต้องคำนวณยอดจ่ายเจ้าของ = รายรับรวมของห้อง/contract − ค่านายหน้า − ค่าใช้จ่ายที่ `responsibilityType = OWNER`
- **FR-051** — ต้องแสดง **ที่มาการหักทุกบรรทัด** (breakdown เป็น line items) ก่อนยืนยันจ่าย
- **FR-052** — ต้องกันจ่ายซ้ำ: ธุรกรรมรายรับ/รายจ่ายหนึ่งรายการเข้าอยู่ใน payout ได้ครั้งเดียว
- **FR-053** — บันทึกการโอน + แนบสลิป + สถานะ payout (`PENDING/PARTIAL/PAID/CANCELLED`) และแสดงในรายงานเจ้าของ

### ตรวจสอบ / อนุมัติ / Audit
- **FR-060** — ทุกธุรกรรม (รายรับ/รายจ่าย/payout) มี `verificationStatus`: `DRAFT/PENDING/VERIFIED/NEEDS_FIX/CANCELLED/PROBLEM`
- **FR-061** — เฉพาะ Admin อนุมัติ (→ `VERIFIED`) ได้ รายการ `VERIFIED` ห้ามแก้/ลบโดยตรง ต้องสร้าง adjustment
- **FR-062** — รายการที่ไม่มีหลักฐานแนบต้องถูก flag เป็น "ต้องตรวจสอบ"
- **FR-063** — ทุก mutation สำคัญต้องเขียน Audit Log (user, action, table, record, old/new, ip, time) และมีหน้าดู/กรอง Audit Log

### Dashboard / รายงาน / Export
- **FR-070** — Dashboard แสดง: รายรับเดือนนี้, รายจ่ายเดือนนี้, กำไรสุทธิเบื้องต้น, ยอดผู้เช่าค้างชำระ, ยอดต้องจ่ายเจ้าของ, ยอดจ่ายเจ้าของแล้ว, ค่าซ่อม/แม่บ้าน, ห้องรายรับสูงสุด/ค่าใช้จ่ายสูงสุด, รายการรอตรวจสอบ, รายการมีปัญหา, รายรับแยกช่องทาง, กราฟรายรับ-รายจ่ายรายเดือน
- **FR-071** — Dashboard/รายงานต้องกรองได้ตาม เดือน/ปี/ช่วงวันที่/ประเภทอาคาร/อาคาร/ห้อง/เจ้าของ/ผู้เช่า/ประเภทรายรับ/ประเภทรายจ่าย/ช่องทาง/สถานะ
- **FR-072** — การคำนวณยอดรวมทั้งหมด **ต้องไม่รวมรายการ `CANCELLED`**; กำไรสุทธิ = รายรับรวม − จ่ายเจ้าของ − ค่าใช้จ่ายที่นายหน้ารับผิดชอบ
- **FR-073** — รายงานหลัก: รายรับรายเดือน, รายจ่ายรายเดือน, แยกเจ้าของ/ห้อง/อาคาร/ผู้เช่า, ยอดค้างรับผู้เช่า, ยอดค้างจ่ายเจ้าของ, ค่าซ่อม/แม่บ้าน
- **FR-074** — Export รายงานเป็น **Excel (.xlsx)** ตามตัวกรองที่เลือก
- **FR-075** — การแจ้งเตือนบน Dashboard: ผู้เช่าค้างชำระ, เจ้าของยังไม่ได้รับเงิน, รายการไม่มีหลักฐาน, รายการรอตรวจสอบ, สัญญาใกล้สิ้นสุด, ห้องว่างนานเกินกำหนด

## 7. Non-functional requirements

- **NFR-Perf-01** — Dashboard โหลด < 3 วินาที บนข้อมูลทั่วไป และรองรับ ≥ 10,000 ธุรกรรมในระยะเริ่มต้น (ต้องมี index ครบ + paginate ทุก list)
- **NFR-Sec-01** — รหัสผ่าน hash ด้วย bcrypt; ป้องกันการเข้าถึงข้ามสิทธิ์ที่ server; ไฟล์แนบ serve แบบ auth-gated (ไม่วางใน `public/`)
- **NFR-Sec-02** — ป้องกัน path traversal/ไฟล์อันตรายตอน upload (ตรวจ mime/นามสกุล/ขนาด, ตั้งชื่อไฟล์ใหม่)
- **NFR-Audit-01** — บันทึก Audit Log ทุกการแก้ไขข้อมูลสำคัญ พร้อมผู้แก้และเวลา
- **NFR-Money-01** — เงินทุกฟิลด์ใช้ทศนิยม fixed-point (`Decimal(12,2)`) ห้ามใช้ float; การรวมยอดต้องแม่นยำ
- **NFR-i18n-01** — UI ภาษาไทยเป็นหลัก; แสดงวันที่เป็น **พ.ศ.**; เก็บวันที่ใน DB เป็น UTC และแปลงที่ UI boundary เท่านั้น
- **NFR-Mobile-01** — ใช้งานได้บนมือถือ/แท็บเล็ต/คอมพิวเตอร์; หน้าบันทึกสลิปถ่ายรูป/อัปโหลดจากมือถือได้สะดวก
- **NFR-Backup-01** — ออกแบบให้ backup ฐานข้อมูล + ไฟล์แนบรายวันได้ (procedure ระดับ ops, ไม่ใช่โค้ดใน v1)

## 8. Success metrics

- **North-star** — % ของธุรกรรมที่มีหลักฐานแนบและสถานะ `VERIFIED` (ยิ่งสูง = ตรวจสอบได้จริง)
- **Leading indicators** — จำนวนรายการ "จ่ายซ้ำ/รับซ้ำ" ที่ระบบดักได้ก่อน commit; เวลาเฉลี่ยในการปิดยอดจ่ายเจ้าของต่อเดือน; จำนวนห้องที่รู้กำไรสุทธิได้ครบ

## 9. Out of scope for v1

ดู Non-goals §3 — สรุป: OCR, import statement, reconciliation, Owner/Tenant portal, LINE notify, invoice/receipt, tax calc, repair ticket, room-availability calendar, เชื่อมแพลตฟอร์มจอง, 2FA, multi-tenant

## Acceptance Criteria (MVP ผ่านเมื่อ)

1. เพิ่มเจ้าของ 1 คน ผูกกับหลายห้องได้
2. เพิ่มอาคารหลายประเภท (คอนโด/บ้านพัก/แฟลต) ได้
3. เพิ่มห้องและกำหนดเจ้าของแต่ละห้องได้
4. เพิ่มผู้เช่าและสร้างรายการเช่าได้
5. บันทึกเงินที่ผู้เช่าจ่าย พร้อมแนบสลิปได้
6. ระบบแสดงสถานะว่าผู้เช่าจ่ายครบหรือยังได้
7. บันทึกค่าแม่บ้าน/ค่าซ่อมผูกกับห้องได้
8. ระบบคำนวณยอดที่ต้องจ่ายเจ้าของได้ (พร้อม breakdown)
9. บันทึกจ่ายเงินเจ้าของ พร้อมแนบสลิปได้ และกันจ่ายซ้ำ
10. Dashboard แสดงรายรับ/รายจ่าย/ยอดค้างรับ/ยอดค้างจ่ายได้
11. กรองข้อมูลตามเดือน/อาคาร/เจ้าของ/ผู้เช่า/ห้องได้
12. Export รายงานเป็น Excel ได้
13. ผู้ใช้แต่ละระดับมีสิทธิ์ต่างกัน (บังคับที่ server)
14. รายการที่อนุมัติแล้วมีประวัติการแก้ไขใน Audit Log
